# Theming

### Theming (semantic, palette-agnostic)

Themes are **single config files**, not code edits. A theme is a `Theme` (`shared/theme/tokens.ts`): `{ id, name, appearance, colors, syntax }`, where `colors` fills the **26 semantic roles** in `COLOR_TOKENS` (backgrounds `--bg`/`--bg-sunken`/`--bg-deep`, surfaces `--surface`/`--surface-2`/`--surface-3`, a text-emphasis ramp `--text`/`--text-secondary`/`--text-muted`/`--text-disabled`/`--text-faint`/`--text-ghost`, and accents/status `--accent`/`--accent-soft`/`--success`/`--warning`/`--warning-soft`/`--danger`/`--info`/`--info-soft`/`--cyan`/`--magenta`/`--cursor`/`--shadow`/`--scrim`/`--input-bg`). **No component references a palette-specific swatch name** (the old `--ctp-mauve`, `--ctp-surface0`, …) — every component uses a `--<token>` semantic role or a composite `--color-*` token built on them, so a new colorscheme is purely data.

There are additionally two **optional roles** (`OPTIONAL_COLOR_TOKENS`), splitting the two jobs `accent` historically did:

- `accent-fill` — the solid accent plane behind primary actions: button fills, active segments, highlighted dialog options, checked checkboxes/toggles. Falls back to `accent`. Indicator dots and gauge fills (status/streaming dots, the context meter) deliberately stay `accent` — they *are* the signal, not a plane under one.
- `on-accent` — text/glyphs painted over `accent-fill`/`accent-soft` fills (button labels, checkmarks, toggle knobs). Falls back to `bg`, exactly what components hardcoded before the role existed (`color: var(--bg)`).

Rationale: a color tuned to be accent *text* on the page background and a color tuned to be a *fill under text* are different jobs — on a black background their luminance windows don't even overlap (AA text needs ≥4.5:1 against black; a fill taking AA light text must stay dim), so a theme must be able to set them independently. With both fallbacks, a theme that sets neither renders pixel-identically to the old single-token behavior; `applyColorScheme` always writes `--accent-fill`/`--on-accent` (theme value or fallback) so a theme switch can't leave stale values. Optional roles must stay optional: making one required would invalidate every pre-existing user theme file at load.

- **Bundled** themes: one pure-data JSON file per colorscheme under `shared/theme/themes/` (`catppuccin-{mocha,macchiato,frappe,latte}.json` + `gruvbox-material-dark.json` + `glow-sticks.json`), each the full `Theme` shape. `bundled.ts` imports them and parses each through `ThemeSchema` so a malformed bundled theme fails loudly at startup. They share the exact on-disk format as user themes, so a brand-new colorscheme is literally one dropped JSON file.
- **User-droppable** themes: `<userData>/themes/*.json` validated against `ThemeSchema` by `main/theme-loader.ts` (invalid/duplicate files skipped, never fatal); exposed via the `themes.listUser` IPC, with the directory path exposed via `themes.userDir` for Settings guidance; the renderer merges them over the bundled set (`theme/registry.ts`, user id wins on collision). The settings picker (`SettingsView`) lists `listThemes()`; `settings.colorScheme` stores the chosen `id` (a free string — `resolveTheme` falls back to the default if a saved id no longer resolves).
- **Application**: `settings-store.applyColorScheme` writes each `theme.colors[role]` as `--<role>` on `:root` and calls `setShikiTheme(theme)`.
- **Syntax highlighting** (`syntax`): either `{ ref: "<shiki-theme-name>" }` (reuse a Shiki-bundled TextMate theme, e.g. `catppuccin-mocha`, `nord`) or `{ inline: <TextMate theme object> }` (ship one — the Gruvbox theme uses this because no Gruvbox theme is bundled with Shiki). `shiki.ts` constructs the highlighter from the concrete object themes (the patched Catppuccin grammars) and then `loadTheme`s each string-ref theme individually with try/catch, so a single unresolvable ref (bundled or user) can never abort highlighting app-wide; it just makes that one theme fall back to the default at render. `setShikiTheme` lazy-loads user refs/inlines the same way. **UI accent tokens can't drive syntax** — this is why `syntax` is its own field, not derived from `colors`.
- **Catppuccin fidelity** (design constraint for the *Catppuccin* themes specifically): their token values stay faithful to the canonical swatches (the only sanctioned deviation is Latte's slightly-lightened surfaces, visible in `themes/catppuccin-latte.json`); depth comes from the surface ramp + hairlines + elevation, accents from translucent real swatches — never invented hues.
- **Catppuccin theming**: Four variants (latte/frappé/macchiato/mocha). Default is mocha. Theme variables set via CSS custom properties.
- **Glow Sticks** (`themes/glow-sticks.json`): OLED-first dark theme — transcript `bg`/`bg-deep` are literal `#000000` and primary text literal `#ffffff` (deep-black panels render those pixels off), with the sidebar plane (`bg-sunken`) deliberately *lighter* than the transcript. The design philosophy: non-text chrome contributes as few photons as possible, so the surface ramp is compressed to just-noticeable steps above black. Accents are verbatim swatches from the [Reasonable Colors](https://www.reasonable.work/colors/) palette built around the orange/aquamarine/indigo triad; the system's shade-3 colors share near-identical luminance (~6.2:1 on pure black), which the status/syntax roles use uniformly. `accent` (text labels, dots, syntax keywords) is the saturated orange-3 at 6.2:1 AA, while `accent-fill` is the dimmer orange-4 (4.2:1, WCAG 1.4.11 UI-component compliant) paired with a warm-white `on-accent`, so primary buttons are light-text-on-ember rather than black-on-blaze — the theme that motivated the role split. Danger is raspberry-3 (not red-3) so it can't be confused with the orange accent, and success is green-3 (hue-distinct from the aquamarine `cyan` role) so diff washes stay unambiguous. The Shiki theme is `inline` (nothing comparable ships with Shiki). Its design contract — pure-black/pure-white anchors, accent contrast floors, ramp ordering, on-accent pairings — is locked by `renderer/src/lib/glow-sticks-theme.test.ts`.
